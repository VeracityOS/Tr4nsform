Section 00 · Context

What We Are Building

The TR4NSFORM Sovereign Peptide Builder (SPB) is a privacy-first, wallet-native web application that allows users to construct personalised peptide protocols, track their health journey, and access TR4NSFORM clinical services — without surrendering a single byte of personally identifiable data to any third party.

The governing principle: The user's wallet IS their account. Their encrypted profile lives on their device — or on infrastructure they personally control. TR4NSFORM never stores, processes, or holds any user health data. Medical professionals access only what the user cryptographically proves on a ZKP basis — nothing more.
Why this matters strategically: This positions TR4NSFORM as the world's first sovereign regenerative medicine platform — a category-defining asset that operates at the intersection of longevity medicine, self-sovereign identity, and Bitcoin-native health infrastructure. Anthony's toolchain must reflect this architectural intent at every layer.
Section 01 · Foundational Principles

Sovereignty Architecture Principles

Eight non-negotiable design laws that govern every technical and UX decision in this system.

🔑
Wallet = Identity
The user's cryptographic wallet address is their sole identity token. No email. No phone. No government ID. No password database. The wallet signs; the system verifies.
🔒
Client-Side Encryption First
All profile data is encrypted on the user's device before it touches any network. The encryption key is derived from the user's wallet signature. We never see plaintext.
🏛
Zero Data at Rest
TR4NSFORM infrastructure holds no user health data, no treatment history, and no personally identifiable information. Servers are stateless conduits, not data warehouses.
⚡
Bitcoin & USDT First
All monetisation flows through Bitcoin Lightning Network and USDT (TRC-20 / ERC-20). No fiat on-ramps that require KYC. No Stripe. No PayPal. No payment processor surveillance.
🧬
ZKP Medical Access Only
Clinical teams access health data exclusively through zero-knowledge proofs. A practitioner can verify that a user meets a protocol threshold without ever seeing the underlying data.
🌐
Non-Custodial Architecture
TR4NSFORM never holds keys, funds, or data on behalf of users. All custody — of identity, of health records, of funds — remains with the individual at all times.
🧩
Modular Sovereignty Layers
Each layer (identity, profile, protocols, payments) operates independently. A user can replace any layer — their own IPFS node, their own Lightning node — without breaking the stack.
👤
Human First
The system serves the individual, not the institution. Clinical professionals are service providers to the user, not custodians of their data. All data rights flow to the patient.
Section 02 · Anti-Requirements

What This System Will Never Do

These are explicit design prohibitions — architectural red lines that cannot be crossed by any feature, sprint, or external pressure.

🚫 Absolute System Prohibitions
  • Collect or store user email addresses
  • Request government-issued identity documents
  • Implement KYC (Know Your Customer) checks of any kind
  • Use third-party analytics that fingerprint users (Google Analytics, Mixpanel, etc.)
  • Store health or biometric data in plaintext on any server
  • Accept fiat payments through KYC-gated processors
  • Integrate social login (Google, Apple, Facebook OAuth)
  • Log IP addresses tied to user session identifiers
  • Share or sell any user data to any third party under any condition
  • Implement server-side session tracking of user behaviour
  • Embed tracking pixels or retargeting scripts
  • Require biometric authentication as a platform prerequisite
  • Store private keys or seed phrases server-side
  • Transmit user health data to medical staff without explicit ZKP user initiation
  • Implement geo-blocking that reveals user location to the platform
  • Require a phone number for any account function
Section 03 · System Architecture

Sovereign Stack Architecture

A layered, privacy-preserving architecture where each layer is independently auditable and replaceable. The user sits at the top; centralised infrastructure is minimised at the base.

👤
User Sovereign Layer
User device. Wallet (MetaMask / WalletConnect / Xverse / BTCPay). Browser-local encrypted storage. User holds master key derived from wallet signature. No data leaves without explicit consent.
Wallet Signature · ECDSA / Schnorr
🔐
Cryptographic Identity Layer
SIWE (Sign In With Ethereum) or Bitcoin message signing. Deterministic key derivation from wallet sig (HKDF). Generates symmetric AES-256-GCM encryption key for profile vault. No username. No password. No seed stored.
AES-256-GCM Encrypted Blob
🧬
Encrypted Profile Vault
User health profile, peptide protocol stack, biometric self-inputs, and treatment history. Stored as encrypted JSON blob. Default storage: IndexedDB (local). Optional: IPFS pinning (user-controlled node). Optional: Arweave permanence.
Decrypted In-Memory Only · Never Plaintext to Network
⚗️
Peptide Builder Application Layer
React SPA (fully client-rendered). Peptide protocol engine, health assessment interface, protocol stack builder, stack visualiser, dosing calculator. All computation client-side. No server required for core functionality.
ZKP Proof Generation · Groth16 / Plonk · snarkjs
🛡
ZKP Clinical Access Layer
User generates a zero-knowledge proof from their encrypted profile proving specific health criteria (e.g., "biomarkers are within range for Protocol X") without revealing underlying data. Proof is transmitted to TR4NSFORM clinical team. Proof is discarded post-verification.
Lightning Invoice / USDT Address · Non-Custodial
⚡
Sovereign Payment Rail
Bitcoin Lightning Network (BTCPay Server / LNbits). USDT TRC-20 / ERC-20. No custodial intermediary. Payment triggers protocol unlock or consultation booking via signed payment receipt — not via identity verification.
Public · Read-Only · No User PII
🌐
Minimal Server Infrastructure
Static file hosting (Vercel / Cloudflare Pages). BTCPay Server for payment verification. Optional IPFS gateway. No databases. No user records. No session storage. All API calls are stateless and PII-free.

Section 07 · Functional Requirements — ZKP Layer

Zero-Knowledge Medical Access Layer

The ZKP layer is the technical implementation of "Human First, Data Never" — allowing clinical verification without data exposure. This is the architectural moat.

ZKP Design Intent: A TR4NSFORM practitioner must be able to confirm that a user's biomarkers support a given protocol without ever receiving, viewing, or storing those biomarkers. The proof is the access token. The data never moves.

FR-ZK · Zero-Knowledge Proof Requirements
IDRequirementSpecificationPriority
FR-ZK-01 Client-Side Proof Generation ZK proof generated entirely in the browser using snarkjs + Groth16 or PLONK proving system. WASM circuit loaded from application bundle. No private inputs ever transmitted to network. Proof generation may take 3–15 seconds depending on device. P1 · High ZKP
FR-ZK-02 Protocol Eligibility Proof Primary proof type: "User's self-reported biomarkers satisfy eligibility criteria for Protocol X." Circuit validates: hormone ranges, renal/hepatic indicators (self-reported), BMI range, contraindication absence. Output: valid/invalid + proof hash. P1 · High ZKP
FR-ZK-03 Age Range Proof User can prove they are within a defined age range (e.g., "I am between 25 and 75") without revealing their exact age or date of birth. Circuit: age ≥ lower_bound AND age ≤ upper_bound. Age input: user self-declared, stored in vault only. P1 · High ZKP
FR-ZK-04 Proof Transmission Package Proof package transmitted to TR4NSFORM clinical endpoint: {proof_hash, public_signals (eligibility booleans only), protocol_id, payment_receipt_hash, timestamp}. No PII. No raw biomarkers. Package signed by user wallet for authenticity. P1 · High ZKP
FR-ZK-05 Clinician Verification Interface Separate TR4NSFORM clinical dashboard (auth-gated for staff). Receives ZKP package. Verifier runs proof check. Displays only: "Protocol X — eligibility CONFIRMED / FAILED" + protocol ID + consult booking link. No health data visible to clinician. P2 · Medium ZKP
FR-ZK-06 Proof Expiry & Discard Proof packages expire 48 hours after generation. Clinical server must verify and discard proofs within this window. No proof data retained after verification. Audit log records only: proof_hash + verification outcome + timestamp. No patient data in log. P1 · High ZKP
FR-ZK-07 Circuit Auditability All ZK circuits published to a public repository (GitHub or IPFS). Circuit source code and compiled artifacts are open-source under AGPL or MIT. Independent audit of circuits commissioned prior to Phase 2 clinical go-live. P2 · Medium ZKP

Section 10 · User Flows

Critical User Journey Architecture

The four primary user journeys that the system must support in Phase 1. All flows begin with wallet connection.

Flow A · First-Time Sovereign Onboarding
Land on Platform
User arrives at tr4nsform.me/build. No sign-up form. Only call to action: "Connect Wallet." Sovereignty manifesto visible above fold.
Connect Wallet
WalletConnect modal opens. User selects wallet. Connection established. Wallet address confirmed. No email requested.
Sign to Unlock Vault
User signs a deterministic message in wallet. Signature derives AES key. Vault is unlocked (empty on first use). Key held in memory only.
Health Self-Assessment
Optional guided assessment. User completes health questionnaire at their own pace. All entries auto-encrypted and saved to local vault. Skip at any time.
Build First Protocol
Protocol engine generates initial recommendations from assessment. User customises stack. Dosing calculator used. Stack saved to vault.
Access Unlocked
User can now access all free-tier features. Premium features prompt payment (Lightning / USDT). No data collected. Journey complete.
Flow B · ZKP Clinical Consultation Request
Request Consultation
User initiates clinical review for their saved protocol stack. System prompts: payment required + ZKP eligibility check required.
Pay via BTC / USDT
Lightning invoice or USDT address generated. User pays. Payment receipt (preimage / TXID) stored in vault. Access token issued locally.
Generate ZK Proof
User selects which health criteria to prove. snarkjs generates proof in browser. 5–15 second process. Private inputs never leave device.
Transmit Proof Packet
Proof + public signals + payment receipt hash + protocol ID transmitted to TR4NSFORM clinical endpoint. No PII. No raw health data.
Clinician Verification
Clinician sees: "Protocol X — ELIGIBLE ✓". No patient data visible. Clinician sends consultation booking link to a user-provided temporary contact (Signal / encrypted email — user chooses).
Proof Destroyed
Clinical server destroys proof packet post-verification. Log records only proof_hash + outcome + timestamp. Consultation proceeds off-platform.
